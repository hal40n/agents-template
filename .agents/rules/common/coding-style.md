# コーディングスタイル

## イミュータビリティ（重要）

常に新しいオブジェクトを生成し、既存のものを直接変更しない：

```text
// 擬似コード
NG:   modify(original, field, value) → originalをその場で変更する
OK:   update(original, field, value) → 変更を加えた新しいコピーを返す
```

理由：イミュータブルなデータは隠れた副作用を防ぎ、デバッグを容易にし、安全な並行処理を可能にする。

## 基本原則

### KISS（シンプルに保つ）

- 実際に機能する最もシンプルな解決策を選ぶ
- 早すぎる最適化を避ける
- 賢さよりも明快さを優先する

### DRY（繰り返しを避ける）

- 繰り返しのロジックは共通関数・ユーティリティに切り出す
- コピペによる実装のズレを避ける
- 抽象化は繰り返しが実際に起きてから行う（投機的な抽象化はしない）

### YAGNI（必要になるまで作らない）

- 必要になる前に機能や抽象化を作らない
- 投機的な汎用化を避ける
- シンプルに始め、必要性が出てきたらリファクタリングする

## ファイル構成

少数の大きなファイルより、多数の小さなファイルを優先する：

- 高凝集・低結合
- 1ファイルあたり200〜400行が目安、最大800行
- 大きなモジュールからはユーティリティを切り出す
- タイプ別ではなく機能・ドメイン別に整理する

## エラーハンドリング

あらゆるレベルで包括的にエラーを処理する：

- エラーはすべてのレベルで明示的に処理する
- UIコードではユーザーフレンドリーなエラーメッセージを表示する
- サーバーサイドは詳細なエラーコンテキストをログに記録する
- エラーを握りつぶさない

## 入力バリデーション

システム境界で必ず検証する：

- すべてのユーザー入力を処理前に検証する
- 利用可能な場合はスキーマベースのバリデーションを使う
- 明確なエラーメッセージで早期にフェイルする
- 外部データを信頼しない（APIレスポンス・ユーザー入力・ファイルの内容）

## 命名規則

- 変数・関数：説明的な名前で `camelCase`（TypeScript / JavaScript）または `snake_case`（Python）
- 真偽値：`is`・`has`・`should`・`can` プレフィックスを優先する
- インターフェース・型・コンポーネント：`PascalCase`
- 定数：`UPPER_SNAKE_CASE`
- カスタムフック（React）：`use` プレフィックス付きの `camelCase`

## 避けるべきコードの臭い

### 深すぎるネスト

ロジックが積み重なり始めたら、ネストした条件分岐より早期リターンを優先する。

### マジックナンバー

意味のある閾値・遅延・上限には名前付き定数を使う。

### 長すぎる関数

大きな関数は責務が明確な小さな単位に分割する。

## コード品質チェックリスト

作業を完了とする前に確認すること：

- [ ] コードが読みやすく、識別子名に意味がある
- [ ] 関数が小さい（50行未満）
- [ ] ファイルが集中している（800行未満）
- [ ] 深いネストがない（4段以下）
- [ ] 適切なエラーハンドリングがある
- [ ] ハードコードされた値がない（定数または設定を使っている）
- [ ] ミュータブルな操作がない（イミュータブルなパターンを使っている）
