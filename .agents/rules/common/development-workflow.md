# 開発ワークフロー

> このファイルは [git-workflow.md](./git-workflow.md) を拡張し、Git操作の前段にあたる機能開発プロセス全体を定義する。

機能実装ワークフローは、リサーチ → 計画 → TDD → コードレビュー → コミットという開発パイプラインを規定する。

## 機能実装ワークフロー

### 0. リサーチと再利用（新規実装の前に必ず行う）

- **まず一次情報を確認する：** 公式ドキュメント・仕様書・ベンダーのAPIリファレンスで動作・パッケージの使い方・バージョン固有の情報を確認してから実装する
- **パッケージレジストリを検索する：** ユーティリティコードを自前で書く前に npm・PyPI などのレジストリを検索する。手製のソリューションより実績あるライブラリを優先する
- **既存の実装を探す：** 問題の80%以上を解決するオープンソースのプロジェクトやコードを探し、移植・ラップできないか検討する
- 要件を満たす実績ある手法を採用・移植することを、ゼロからの実装より優先する

### 1. 計画を立ててから実装する

- 複雑な機能は実装前にアプローチを整理する
- 依存関係とリスクを特定する
- フェーズに分解してから着手する
- 必要に応じて設計内容を `.agents/memories/` にメモしておく

### 2. テスト駆動開発（TDD）

- **`tdd-guide`** に従って進める
- テストを先に書く（RED）
- テストが通る最小限の実装を書く（GREEN）
- リファクタリングする（IMPROVE）
- カバレッジ80%以上を確認する

### 3. コードレビュー

- コードを書いた直後に **`code-reviewer`** を使う
- CRITICALおよびHIGHの問題は必ず対処する
- MEDIUMの問題は可能な限り修正する
- Pythonコードは **`python-reviewer`** も使う

### 4. コミットとプッシュ

- 詳細なコミットメッセージを書く
- コンベンショナルコミット形式に従う
- コミットメッセージとPRプロセスの詳細は [git-workflow.md](./git-workflow.md) を参照

### 5. マージ前チェック

- 自動チェック（CI/CD）がすべて通過していることを確認する
- マージコンフリクトを解消する
- ブランチがターゲットブランチと同期されていることを確認する
- これらのチェックが通過してからレビューを依頼する
