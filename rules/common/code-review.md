# コードレビュー基準

## 目的

コードレビューは、マージ前に品質・セキュリティ・保守性を確保するためのプロセス。このルールはレビューのタイミングと進め方を定義する。

## レビューのタイミング

**必須のレビュートリガー：**

- コードを作成・変更した後
- 共有ブランチへのコミット前
- セキュリティに関わるコードを変更したとき（認証・決済・ユーザーデータ）
- アーキテクチャに関わる変更を加えたとき
- プルリクエストのマージ前

**レビュー前の前提条件：**

レビューを依頼する前に以下を確認すること：

- 自動チェック（CI/CD）がすべて通過している
- マージコンフリクトが解消されている
- ブランチがターゲットブランチと同期されている

## レビューチェックリスト

コードを完了とする前に確認すること：

- [ ] コードが読みやすく、識別子名に意味がある
- [ ] 関数が集中している（50行未満）
- [ ] ファイルが凝集している（800行未満）
- [ ] 深いネストがない（4段以下）
- [ ] エラーが明示的に処理されている
- [ ] シークレットや認証情報がハードコードされていない
- [ ] `console.log` やデバッグ文が残っていない
- [ ] 新機能にテストがある
- [ ] テストカバレッジが最低80%を満たしている

## セキュリティレビューのトリガー

**以下に該当する場合は作業を止めて `security-reviewer` を使うこと：**

- 認証・認可のコード
- ユーザー入力の処理
- データベースクエリ
- ファイルシステム操作
- 外部API呼び出し
- 暗号化処理
- 決済・金融に関わるコード

## レビューの深刻度レベル

| レベル | 意味 | アクション |
| --- | --- | --- |
| CRITICAL | セキュリティ脆弱性・データ損失リスク | **ブロック** — マージ前に必ず修正 |
| HIGH | バグ・重大な品質問題 | **警告** — マージ前に修正すべき |
| MEDIUM | 保守性の懸念 | **情報** — 修正を検討する |
| LOW | スタイル・軽微な提案 | **メモ** — 任意 |

## エージェントの使い分け

| エージェント | 目的 |
| --- | --- |
| **code-reviewer** | 全般的なコード品質・パターン・ベストプラクティス |
| **security-reviewer** | セキュリティ脆弱性・OWASP Top 10 |
| **python-reviewer** | Python固有の問題 |

## レビューワークフロー

```text
1. git diff で変更内容を把握する
2. セキュリティチェックリストを最初に確認する
3. コード品質チェックリストを確認する
4. 関連するテストを実行する
5. カバレッジが80%以上であることを確認する
6. 適切なエージェントで詳細レビューを行う
```

## よくある問題

### セキュリティ

- ハードコードされた認証情報（APIキー・パスワード・トークン）
- SQLインジェクション（クエリへの文字列連結）
- XSS脆弱性（ユーザー入力のエスケープ漏れ）
- パストラバーサル（サニタイズされていないファイルパス）
- CSRF保護の欠如
- 認証バイパス

### コード品質

- 大きすぎる関数（50行超）→ 小さく分割する
- 大きすぎるファイル（800行超）→ モジュールに切り出す
- 深すぎるネスト（4段超）→ 早期リターンを使う
- エラーハンドリングの欠如 → 明示的に処理する
- ミュータブルな操作 → イミュータブルな操作を優先する
- テストの欠如 → テストカバレッジを追加する

### パフォーマンス

- N+1クエリ → JOINやバッチ処理を使う
- ページネーションの欠如 → クエリに LIMIT を追加する
- 無制限クエリ → 制約を追加する
- キャッシュの欠如 → コストの高い処理をキャッシュする

## 承認基準

- **承認**：CRITICALおよびHIGHの問題がない
- **警告**：HIGHのみ（注意してマージ可）
- **ブロック**：CRITICALが見つかった場合
