# write-readme

プロジェクトの README.md を執筆・更新するためのワークフローを定義するスキル。
利用者が初見で「何ができるか」「どう使うか」を最短で理解できる README を目指す。

---

## 起動条件

以下の発言・コマンドで自動的にこのスキルを使う：

- 「READMEを書いて」「READMEを作って」「README更新して」
- `/write-readme`
- 新規プロジェクト直下に `README.md` が存在しない or 空のときの執筆依頼

---

## ワークフロー

### 1. プロジェクトの自己観察（既存ファイルから推測する）

ユーザーに質問する前に、可能な範囲でプロジェクト情報を**自動収集**する。

| 取得対象 | 取得元 |
| --- | --- |
| プロジェクト名 | ディレクトリ名・`package.json` の `name`・`pyproject.toml` の `name` |
| 言語/フレームワーク | `package.json`・`pyproject.toml`・`Cargo.toml`・`go.mod`・主要拡張子 |
| エントリポイント | `package.json` の `main`/`bin`/`scripts`・`src/index.*`・`main.*` |
| 依存関係 | 同上のマニフェストファイル |
| ライセンス | `LICENSE` ファイル・`package.json` の `license` |
| バージョン | マニフェストファイル・`git tag` |
| テスト/ビルド方法 | `package.json` の `scripts`・`Makefile`・`pyproject.toml` の scripts |
| GitHub情報 | `git remote -v` の URL |
| 既存README | `README.md`（更新時）・`docs/` 配下 |

**ツール呼び出しは並列で実行する**：
- `Read package.json`（または `pyproject.toml` 等）
- `git remote -v`、`git log --oneline -1`
- `LS` でディレクトリ構成把握
- 既存 `README.md` の `Read`

### 2. ユーザーへの確認（不明点のみ）

自動収集で埋まらない項目だけを `AskUserQuestion` で聞く：

1. **プロジェクトの目的・解決する課題**（1〜2文）
2. **想定する利用者**（OSS公開 / チーム内 / 個人用）
3. **README に含める範囲**（下記セクション選択）
4. **強調したいセールスポイント**（任意）

不明点がなければスキップして次へ進む。

### 3. セクション構成の提案

プロジェクトタイプ別の**推奨構成**を提示し、ユーザーに確定させる。

#### A) OSSライブラリ / npm パッケージ

```PlainText
1. プロジェクト名 + バッジ（CI / npm version / license / coverage）
2. 一言サマリー（hero description）
3. 特徴（Features）
4. インストール（Installation）
5. クイックスタート（Quick Start）
6. 使い方（Usage）
7. API リファレンス概要
8. 設定（Configuration）
9. 開発者向け情報（Development）
10. コントリビューション（Contributing）
11. ライセンス（License）
```

#### B) CLIツール

```PlainText
1. プロジェクト名 + バッジ
2. 一言サマリー
3. インストール
4. 使い方（コマンド例 + 出力例）
5. オプション一覧
6. 設定ファイル（あれば）
7. ライセンス
```

#### C) Webアプリケーション（個人/SaaS）

```PlainText
1. プロジェクト名
2. 概要 + スクリーンショット
3. 主要機能
4. 技術スタック
5. セットアップ（環境変数・依存・DB等）
6. 開発用コマンド
7. デプロイ
8. ライセンス
```

#### D) 個人開発・実験プロジェクト

```PlainText
1. プロジェクト名 + 一言サマリー
2. 何をするものか / なぜ作ったか
3. セットアップ
4. 使い方
5. 既知の制約・TODO
```

### 4. 執筆ルール

#### 言語

- **日本語で執筆する**（コード例・コマンド例は英語のまま）
- OSS公開を想定する場合のみ、ユーザーに確認の上で英語版も併設

#### 文体

- 一文を短く、見出しを多めに切る
- 「〜です／〜ます」で統一（または「〜だ／〜である」をプロジェクト方針に合わせる）
- 主観形容詞（素晴らしい・完璧・革新的等）を使わない
- 動詞を主役にする（"提供する" より "返す・出力する・生成する"）

#### 必須のコードブロック

すべてのインストール・使い方の例は**実行可能な形**で記載する。

````markdown
```bash
npm install <package-name>
```

```typescript
import { foo } from '<package-name>';

const result = foo({ input: 'hello' });
console.log(result);
```
````

#### バッジ

OSS の場合は以下を最初に並べる（実在するもののみ）：

- CI ステータス：`![CI](https://github.com/<owner>/<repo>/actions/workflows/<file>/badge.svg)`
- npm バージョン：`![npm](https://img.shields.io/npm/v/<package-name>.svg)`
- ライセンス：`![License](https://img.shields.io/badge/license-<MIT等>-blue.svg)`
- カバレッジ：`![Codecov](https://codecov.io/gh/<owner>/<repo>/branch/main/graph/badge.svg)`

存在しないものを書かない。

#### 構成のルール

- **TOC（目次）はセクション数が10以上の場合のみ**追加する
- 画像は `docs/images/` または `assets/` に置き、相対パスで参照する
- 内部リンクには `#見出し` を使い、英数字ハイフンに正規化される点に注意する
- 末尾に「## License」を必ず置く（個人用でない限り）

### 5. ドラフト生成

確定した構成に従って README.md を執筆する。

- **既存ファイルがある場合**：差分上書きの確認を取ってから書き込む（CLAUDE.md の安全ルール準拠）
- **新規の場合**：そのまま `Write` で作成する
- 不明な箇所は `<TODO: ...>` で明示する。ユーザーが後で埋められるようにする

### 6. 仕上げチェック（必須）

書き終わったあと、以下を自己レビューする：

- [ ] プロジェクト名・パッケージ名・コマンド名が**実在する**ものと一致しているか
- [ ] インストール手順を**コピペで実行**できるか
- [ ] クイックスタートが**5分以内**で動かせる粒度か
- [ ] バッジのURLが**現実に通る**か（存在しないリポジトリ/CIを書かない）
- [ ] ライセンスが `LICENSE` ファイルと一致しているか
- [ ] 外部リンクが切れていないか（必要に応じて WebFetch で確認）
- [ ] 個人情報・APIキー・内部URLが含まれていないか

問題があれば修正してから完了報告する。

---

## 禁止事項

- 存在しない機能・未実装の機能を「使い方」セクションに書かない
- 「素晴らしい」「完璧な」「革新的な」等の主観的形容詞を使わない
- 絵文字を使わない（ユーザーが明示的に許可した場合のみ）
- 大量の宣伝文句・ポエム的な導入を入れない
- ユーザーに確認せず英語版に翻訳しない

---

## 完了報告フォーマット

執筆後、以下を簡潔に報告する：

```PlainText
README.md を作成しました（<行数>行 / <セクション数>セクション）。

主なセクション:
- <セクション1>
- <セクション2>
- ...

未確定の項目（<TODO>として残しています）:
- <項目>

次にやると良いこと:
- LICENSE ファイル追加（未配置の場合）
- スクリーンショット配置（Webアプリの場合）
```
